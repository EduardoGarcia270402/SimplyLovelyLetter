1. OBJETIVO DEL SPRINT

Diseñar e implementar la capa de modelos (models/) del backend
utilizando Mongoose, definiendo las entidades principales del dominio
del sistema.

Este sprint se enfoca exclusivamente en la estructura de datos,
relaciones, validaciones y configuración de persistencia en MongoDB.

No se implemento lógica de negocio ni controladores en esta fase.

2. Arquitectura
Se crearon tres modelos principales:
* User
* Template
* Letter
Tambien se creo un archivo index.js para centralizar las exportaciones
de los modelos.

2.1 MODELO 1: USER

Representa al padre, tutor o administrador que utiliza la plataforma.

Campos principales: 
- name - email (único e indexado) 
- password (con hash automático) 
- role (parent | admin) 
- timestamps (createdAt,updatedAt)

Características técnicas: 
- Validaciones a nivel esquema 
- Índice único en email 
- Middleware pre(‘save’) para hash de contraseña 
- Método de instancia comparePassword 
- password con select: false por seguridad

¿Porque se hizo asi?: El sistema requiere autenticación y control de acceso.
Este modelo permite escalar hacia un panel administrativo en el futuro.

2.2 MODELO 2: TEMPLATE
Representa las plantillas visuales que los niños pueden seleccionar para
personalizar sus cartas digitales.

Campos principales: 
- name 
- previewImage (URL) 
- backgroundColor 
- isActive 
- timestamps

¿Porque se hizo eso?: para separar las plantillas como entidad independiente
permite: - Activar o desactivar plantillas sin eliminarlas - Escalar a
un catálogo administrable - Mantener normalización en la base de datos

2.3 MODELO 3: LETTER
Entidad central del sistema. Representa la carta digital creada.

Campos principales: 
- childName 
- content (texto transcrito desde voz) 
- audioUrl (opcional) 
- template (referencia a Template) 
-recipientEmail 
- status (draft | sent | failed) 
- user (referencia a User) 
- timestamps

Relaciones: 
- Un User puede tener múltiples Letters (1:N) 
- Un Template puede ser usado en múltiples Letters (1:N)

¿Porque sehizo asi?: Este modelo concentra la lógica principal del dominio.
Permite controlar el flujo de envío, almacenar historial y mantener
relaciones consistentes entre usuarios y plantillas.

3.DISEÑO Y DECISIONES ARQUITECTÓNICAS

-   Arquitectura modular con un archivo por modelo.
-   Validaciones estructurales en la capa de persistencia.
-   Uso de referencias ObjectId para relaciones normalizadas.
-   Preparado para autenticación JWT en próximos sprints.
-   Preparado para integración con almacenamiento en la nube para audios
    (ej. S3 o Cloudinary).
-   Uso de timestamps para auditoría automática.

Este sprint deja el backend listo para construir la capa de acceso a
datos (repositories) en el Sprint 3.
